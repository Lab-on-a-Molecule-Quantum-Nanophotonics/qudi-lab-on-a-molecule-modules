Basics of Qudi

Why automation?

We operate complex instruments and perform repetitive measurements: computers are a good help for that. We want softwares that:

  • Is able to handle the variability in instrumentation,
  • Facilitates the systematic handling of experiments,
  • Provides traceability of data provenance and experimental settings,
  • Is easy enough to tweak, expand, and re-use.

Why a framework?

We want to limit how much we code

  • This is time consuming,
  • Someone likely already did that,
  • We are not software engineers.

A good software:

  • performs its task correctly,
  • performs its task efficiently,
  • is documented so users can learn how to use it and understand what it does,
  • has public, documented code so new people can work on it,
  • has sufficiently well structured code that it will facilitate maintenance and addition of new functionalities.

A framework allows us to re-use some well tested solutions to these problems.

Qudi

Qudi has been developped at Ulm-IQO, by people working on NV center in diamond. It is much broader in scope!

Please cite their original paper on Qudi if you use it.

The Hardware, Logic, GUI stack

Logic modules

  • The brain of you experiment.
  • Orchestrate hardwares and other logics to acquire data, analyze, and save them.

Hardware modules

  • An abstraction layer for Qudi to use the instrument.
  • The way to interact with them is standardized by an Interface.

Interfaces

  • A template to write a hardware module.
  • Example: the laser scanning logic knows how to use an ExcitationScannerInterface, so the laser hardwares must implement it.

GUI

  • What the user sees.
  • It is usually logic-specific.
  • The logic knows nothing about it
  • Separation of concern!

  • startup_modules: This lists the modules we want to start automatically when we run Qudi. This avoids having to launch the manually.
  • remote_modules_server: This is how we make Qudi aware of another instance running on another network. Refer to the documentation for how to use it.
  • default_data_dir: This is where Qudi will store the data when saving. We systematically overwrite this in our configurations to make it point to the shared drives!

Modules section


                                logic:

                                        options:
                                            my_config_option: 'I am a string from the Qudi configuration'
                                        connect:
                                            a_connector: a_defined_hardware_or_logic
  • module.Class: Which Python class we want to use.
  • Connectors: the connect section tells Qudi which module in the configuration it should link to this module. You know what is expected thanks to the documentation/code.
  • Configuration options: the options section gives static configuration values for the module.

Level 1: a logic module

Our laser seems to have an offset in frequency, let's write an optimizer! Using the provided template, write the FrequencyOptimizeLogic code. The function should

  1. Scan specified range of frequencies using the _scanner connector
  2. Find the maximum of the data, its frequency f0, and the FWHM.
  3. Scan step by step a region [f0-6FWHM, f0+6FWHM] by setting manually the idle frequency of the excitation_scanner_logic. For each of these steps, record the fluorescence by probing the "APD" channel of the process_control module.
  4. Set the _scanner idle frequency to the frequency of the actual maximum.

Hints

  • Don’t forget the parentheses when calling the connectors: self._scanner().something and not self._scanner.something
  • Numpy has many useful functions, such as argmax
  • Do read the code in src/qudi/logic/scanning_excitation_logic.py and src/qudi/hardware/tutorial/process_control.py to understand how to use them! There are things like acquisition_running and get_spectrum that should catch your eye.
  • Remember to restart the jupyter kernel each time you restart Qudi.

Configuring, Starting a scan and waiting for completion


                        self._scanner().set_variable("Start frequency", fstart)
                        self._scanner().set_variable("Stop frequency", fstop)
                        self._scanner().set_variable("Step frequency", fstep)
                        self._scanner().get_spectrum()
                        # wait for the spectrum to finish
                        while self._scanner().acquisition_running:
                            time.sleep(0.5)

Finding the central frequency and fwhm


                        spectrum = self._scanner().spectrum
                        frequency = self._scanner().frequency
                        bg = np.quantile(spectrum, 0.1)
                        imaxi = np.argmax(spectrum)
                        amplitude = spectrum[imaxi] - bg
                        f0 = frequency[imaxi]
                        istart = np.nonzero(np.logical_and(spectrum - bg < amplitude/2, frequency < f0))[0][-1]
                        istop = np.nonzero(np.logical_and(spectrum - bg < amplitude/2, frequency > f0))[0][0]
                        fstart = frequency[istart]
                        fstop = frequency[istop]
                        fwhm = abs(fstop - fstart)

Manual scan and optimization of laser frequency


                        # 3. Manually scan the frequencies
                        scanned_setpoints = np.linspace(f0-6*fwhm, f0+6*fwhm, 100)
                        fluorescence = np.zeros_like(scanned_setpoints)
                        for (i,f) in enumerate(scanned_setpoints):
                            self._scanner().idle = f
                            time.sleep(0.1)
                            fluorescence[i] = self._process().get_process_value("APD")
                            imax = np.argmax(fluorescence)
                        # 4. Go there
                        self._scanner().idle = scanned_setpoints[imax]
